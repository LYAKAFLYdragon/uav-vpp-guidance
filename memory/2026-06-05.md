## Stage 6G.5D Completion (2026-06-06)

- **Critical bug found and fixed**: `mode_switch_pn_no_vpp` was 0/90 despite gate activating.
- **Root cause**: Mode-switch gate active on step 1 (aspect=14°), but deactivated on step 2 (aspect grew to 17.9° > 15° threshold). When gate deactivated, code fell back to VPP policy virtual point + primary guidance, causing divergence. The VPP offset (~500m norm) pushed the virtual point away from target.
- **Fix**: Added episode-level latch `self._mode_switch_latched` in `CloseRangeTrackingEnv`:
  - Once gate activates, latch stays True for entire episode.
  - `mode_switch_reason` = `"latched"` when gate would otherwise be inactive.
  - Latch resets on `env.reset()`.
  - Also fixed missing `self._guidance_pn.reset()` in `env.reset()` (would have caused stale LOS filter state across episodes).
- **File changes**:
  - `src/uav_vpp_guidance/envs/tracking_env.py`: latch logic + reset fix.
  - `tests/test_stage6g5d_pn_mode_switch.py`: added `TestModeSwitchLatch` with 2 tests.
- **Results after fix** (3 points × 3 seeds × 10 episodes = 90 per variant):
  - `pure_pn_no_vpp`: 90/90 SUCCESS (unchanged baseline)
  - `vpp_policy_pn_guidance`: 0/90 crash (unchanged, VPP offset harmful)
  - `mode_switch_pn_no_vpp`: **90/90 SUCCESS** (was 0/90 before fix)
  - `mode_switch_vpp_elsewhere`: **90/90 SUCCESS** (VPP+LOS normally, latched PN in gate)
- **Interpretation**:
  - The mode-switch latch successfully rescues VPP-based architectures by bypassing VPP offset when geometry demands direct pursuit.
  - `mode_switch_vpp_elsewhere` proves we can keep VPP for non-tail-chase geometries while using PN for tail-chase.
